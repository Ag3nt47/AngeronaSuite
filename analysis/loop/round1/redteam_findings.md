# Round 1 ‚Äî Red Team Findings

Scope: `src/angerona/**`, root `*.bat`/`*.ps1`, `.gitignore`. Read-only audit.
Deployment model assumed: single Windows host, one elevated operator. Severity =
impact √ó exploitability in that model, existing mitigations credited.

Bottom line: A-01/A-02/A-03/A-05 verified still resolved. The interesting new
result is that the **exact control A-03 added for `cve_fix_advisor` (a destructive
denylist scan on AI-generated PowerShell) is absent in two *other* AI->PowerShell
execution paths that were not covered by A-03** -- a newer SOAR component
(`shark/playbook_tuner.py`) and `posture_hardening.execute_remediation`. Those are
filed as new findings rather than re-raising A-03.

## R1-01 -- LLM-generated PowerShell wired into the auto-executed SOAR gate with no denylist / review gate
- **Severity:** MEDIUM
- **Component:** `src/angerona/shark/playbook_tuner.py:37` (`_ollama_block`), `:114` (`pb.write_text`), `:78-88` (`_register_in_gate`); executed via root `mitigation_gate.ps1` (dot-source) -> `src/angerona/engines/defense_monitor.py:74` / manual operator run. Trigger: `src/angerona/modules/posture_hardening.py:519-520`.
- **Description:** On a containment bypass (Judgment `VERIFICATION_RESULT: SUCCESS`),
  `posture_hardening` calls `tune_containment(technique_id)`. That function asks the
  local Ollama model for "ONLY raw PowerShell" (`_SYS`, temperature 0), takes the raw
  model response (`_ollama_block`), strips fences, and writes it verbatim to
  `playbooks/dynamic_block_<tid>.ps1` (`pb.write_text(header + block)`), then
  idempotently appends a `. "$PSScriptRoot\playbooks\<name>.ps1"` dot-source line into
  the repo-root `mitigation_gate.ps1` (`_register_in_gate`). `mitigation_gate.ps1` is
  the SOAR containment gate that runs elevated with `powershell.exe -ExecutionPolicy
  Bypass` (invoked by `defense_monitor.trigger_mitigation_gate`, and intended for
  operator/SOAR execution). Unlike the sibling path fixed under A-03
  (`cve_fix_advisor.scan_powershell`), there is **no destructive-content denylist and
  no per-run human review or hash-integrity gate** between the model output and its
  eventual elevated execution. The docstring claims "nothing auto-executes from here,"
  which is true only inside `tune_containment` itself -- the generated bytes are
  persisted into a gate that *does* auto-execute later.
- **Impact:** A prompt-injected, poisoned, or simply hallucinating local model can emit
  destructive PowerShell (e.g. `Remove-Item -Recurse`, `Set-MpPreference
  -DisableRealtimeMonitoring $true`, account/firewall changes) that is written into the
  containment gate and then runs with Administrator rights the next time the gate fires.
  No content check would stop it.
- **Exploitability notes (why MEDIUM not HIGH):** requires the drill/judgment loop to
  reach a real containment bypass to trigger; the prompt `timeline` is developer-built
  (no direct external-attacker string reaches the model here); the model is local Ollama;
  and `defense_monitor`'s hard-coded gate path is `engines\mitigation_gate.ps1` (a path
  mismatch vs the repo-root gate the tuner edits), so the most likely execution is a
  manual/operator run of the root gate. These reduce likelihood but not the missing-control gap.
- **Recommendation (remediation agent):** Reuse `core.cve_fix_advisor.scan_powershell`
  (the A-03 destructive denylist) inside `playbook_tuner.tune_containment` -- scan
  `block` before `pb.write_text`, and refuse to write / register any block that trips a
  destructive pattern (fall back to `_fallback_block`, which is deterministic and safe).
  Optionally stamp a hash and require the same review gate `posture_hardening` uses.

## R1-02 -- `posture_hardening.execute_remediation` runs AI-generated PowerShell with no A-03 destructive denylist
- **Severity:** MEDIUM
- **Component:** `src/angerona/modules/posture_hardening.py:356-366` (`_generate_remediation` via `_SYS_REMEDIATE`/`_ollama`), `:427-473` (`execute_remediation` -> `_run_powershell_file` -> `powershell -ExecutionPolicy Bypass -File`). GUI callers: `src/angerona/gui/pages.py:704`, `src/angerona/gui/main_window.py:842`.
- **Description:** `_generate_remediation` asks Ollama to author a hardening PowerShell
  script and stages it as `<mitre>.ps1`. `execute_remediation(authorized=True)` then
  executes those bytes elevated. The path has a genuine TOCTOU-closed hash-integrity
  check and an `authorized` flag, but -- unlike the parallel `cve_fix_advisor` path fixed
  in A-03 -- it applies **no destructive-content denylist** to the model output before
  running it. The GUI confirmation in `pages.py:_apply_fix` warns the user but does not
  force display/diff of the exact script, and the AAR-dialog bulk `_apply`
  (`main_window.py:837-846`) loops over every vulnerable weakness calling
  `execute_remediation(..., authorized=True)` with no per-script confirmation at all.
- **Impact:** Same class as R1-01 -- a poisoned/hallucinating local model can produce a
  destructive staged script that passes the integrity check (it is hashed *as generated*)
  and runs as Administrator on operator authorization, with nothing scanning for
  destructive verbs.
- **Exploitability notes:** Requires operator to click Apply / confirm; local model only;
  hash gate prevents on-disk tampering but not malicious *generation*. Hence MEDIUM.
- **Recommendation:** Run `scan_powershell` (A-03 denylist) on the generated script both
  at `_generate_remediation` (refuse to stamp/stage a destructive script) and inside
  `execute_remediation` immediately before `_run_powershell_file`. Make the AAR-dialog
  bulk `_apply` require the same per-script confirmation the single-fix path uses.

## R1-03 -- SOAR gate + playbooks directory is a writable trust boundary (potential local privilege escalation)
- **Severity:** LOW  (MEDIUM if the install dir is world/user-writable)
- **Component:** root `mitigation_gate.ps1` (dot-sources `playbooks\*.ps1`), written/extended by `src/angerona/shark/playbook_tuner.py:78-88,114`; executed elevated via `src/angerona/engines/defense_monitor.py:74`.
- **Description:** `mitigation_gate.ps1` lives in the repo root and statically dot-sources
  `playbooks\dynamic_block_*.ps1`; both are executed with `-ExecutionPolicy Bypass` at
  Administrator. Angerona is deployed from a non-`Program Files` location
  (e.g. `<install-folder>`). Drive roots frequently
  inherit ACLs that grant **Users: Write**. If that is the case here, any non-admin local
  user can modify `mitigation_gate.ps1` or drop a `playbooks\dynamic_block_*.ps1`, and
  their code will run as Administrator the next time the SOAR gate fires -- a classic
  writable-script LPE. This is distinct from A-04 (which covers the Python drop-in module
  dir) -- this is the PowerShell playbook dir + repo-root gate script.
- **Impact:** Local privilege escalation to the elevated Angerona/SOAR context.
- **Recommendation:** Before running the gate, verify the gate script and `playbooks\`
  directory are not writable by non-admins (deny/log if they are), or relocate the runtime
  gate + generated playbooks under a per-user protected data dir (e.g. `%LOCALAPPDATA%\
  Angerona\playbooks`) whose ACL Angerona controls, and reference that path from a gate
  Angerona installs with a restrictive ACL. Ties into A-06 (centralize/allowlist/log PS).

## R1-04 -- Dead legacy sniffer leaks observed remote IPs to a third party over plaintext HTTP
- **Severity:** INFO
- **Component:** `src/angerona/engines/sniffer.py:59-60` (`http://ip-api.com/json/{ip}`), `:48` (starts a DPI worker thread at import).
- **Description:** `engines/sniffer.py` sends every observed non-local remote IP to
  `http://ip-api.com` over cleartext HTTP for geolocation and spins up a DPI worker thread
  on import. No module imports it (the live "Packet Sniffer" is the unrelated
  `modules/packet_sniffer.py`), so it is currently dead code -- but if it is ever wired in
  it would exfiltrate host connection metadata unencrypted and could be MITM-influenced.
- **Impact:** None while unimported; privacy/metadata leak if activated.
- **Recommendation:** Delete `engines/sniffer.py` (dead), or if kept, switch geolocation to
  HTTPS and gate it behind explicit opt-in like the other off-host modules
  (`remote_bridge`, `siem_forwarder`).
